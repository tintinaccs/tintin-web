CREACIÓN (reglas de Firestore + transacción)
- qty entero 1..99, precio >= 0, 1..4 ítems distintos, subtotal derivado de las
  líneas y total == subtotal + envío. Bloquea cantidades inválidas y totales
  negativos ANTES de escribir. Tienda abierta, cuenta no bloqueada y correo
  verificado son obligatorios.

EDICIÓN COMPLETA (saveOrderEdit)
- Permiso manageOrdersFull + editarCompleto. Valida nombre y teléfono. Rechaza
  cantidades < 1 y total < 0. El total se DERIVA (subtotal + envío), usando el
  precio HISTÓRICO guardado en cada ítem (nunca re-consulta el producto), así el
  pedido conserva sus montos aunque el producto luego cambie o se borre. No deja
  quitar el último producto. Anti-doble-guardado (btn.disabled). Registra
  editar_pedido en Auditoría. Recuperación ante error.

ESTADOS / PAGO / ELIMINACIÓN / MASIVAS
- updateOrderStatus / updatePayStatus: permiso + auditoría + recuperación (si
  falla, repone el valor real). Estados provienen de <select> fijos → no hay
  estados imposibles. Correo automático opcional fire-and-forget (nunca bloquea
  ni revierte el cambio).
- deleteOrder: permiso manageOrdersFull + confirmación destructiva + auditoría.
- Acciones masivas: permiso accionesMasivas + el permiso puntual (estado/pago) +
  auditoría con conteo.
- Reenvío: cooldown fijo de 60 s por pedido + permiso reenviarCorreo + límite
  diario del lado servidor (Apps Script).

CONSISTENCIA ENTRE SUPERFICIES
- El pedido es AUTOCONTENIDO: guarda cliente (nombre/tel/email), envío (ciudad/
  dirección/referencia/mapa/método), pago (método/estado), ítems con su precio,
  subtotal, envío, total, notas y timestamps. Alias heredados (city/address/
  paymentStatus/shippingMethod) se leen con fallback.
- Perfil del cliente lee o.items/o.total guardados (escapados), no el producto
  en vivo. Estadísticas suman con Math.max(0, ...) → nunca un total negativo.
- WhatsApp del cliente en el panel usa o.userPhone/o.userName del pedido.

HALLAZGO CORREGIDO
═══════════════════════════════════════════════════════════════════════
F1 [MEDIO] — Conflicto entre administradores al editar el mismo pedido.
  saveOrderEdit hacía un updateDoc "ciego" con el snapshot tomado al abrir el
  editor. Si un segundo administrador cambiaba el pedido (estado, nota, etc.)
  mientras el primero tenía el editor abierto, al guardar se pisaban sus cambios
  en silencio (lost update). El btn.disabled solo evitaba el doble clic del mismo
  administrador, no el conflicto entre dos.
  FIX (js/admin-app.js) — concurrencia optimista, FAIL-OPEN:
    - openOrderEdit guarda _orderEditBaselineMillis = updatedAt real del pedido.
    - saveOrderEdit, antes de escribir, relee el pedido (getDoc) y:
        · si ya no existe → avisa (no lo recrea);
        · si su updatedAt es más nuevo que la base → aborta con aviso claro para
          que el admin cierre y reabra sin pisar cambios ajenos.
    - Si por cualquier motivo no se puede comparar (sin updatedAt, error de red),
      se guarda igual (fail-open) para no bloquear una edición legítima.
    - closeOrderEdit resetea la base.

Evaluado y descartado como cambio (para no fabricar arreglos ni cambiar
comportamiento intencional):
- Restaurar stock al eliminar/cancelar un pedido: la creación descuenta stock en
  la transacción, pero eliminar/cancelar NO lo repone. Es una simplificación
  deliberada del plan gratuito (el stock se ajusta manualmente); reponerlo
  requeriría otra transacción con casos borde (producto borrado, stock ya
  ajustado a mano). Se documenta como comportamiento conocido, no se toca.
- Recalcular el costo de envío al cambiar método/ciudad en el editor: el modal no
  expone un campo de costo de envío (se conserva el del checkout); no es un bug
  visible. Documentado.
