REVISADO Y CONFIRMADO YA CORRECTO (sin cambios)
═══════════════════════════════════════════════════════════════════════
CONTENIDO
- Firestore guarda valores, nunca HTML/CSS/selectores (los selectores viven en
  content-schema.js). El sitio público aplica cada valor a su selector real
  (querySelectorAll(item.selector)) → cada control produce un cambio real.
- Render seguro: createTextNode + replaceChildren + appendPlainLines; jamás
  innerHTML de datos de Firestore. Los enlaces pasan por sanitizeContentHref
  (bloquea javascript:/data:/vbscript:/file:) antes de setAttribute('href').
- onSnapshot para reflejar cambios en vivo; campos ausentes conservan el HTML
  publicado (fallback).
- Reglas: site_content acotado a Super Admin o al permiso dinámico de contenido.

APARIENCIA
- Cada token del catálogo mapea a una variable CSS real (--color-*) que un
  consumidor usa (verificado por audit-color-scheme) → sin opciones decorativas.
- Los colores se aplican como custom properties (root.style.setProperty valor
  puro), nunca como reglas/bloques CSS.
- Preview inmediato sin mutar el borrador (aparTransientColor / onPreview),
  guardia de cambios sin guardar (aparHasPending / aparRegisterUnsavedGuard),
  overrides por dispositivo (DEVICE_BREAKPOINTS + deviceOverrides), y
  sincronización en tiempo real (onSnapshot sobre colorSchemes y settings/appearance).
- Ante un error de lectura del esquema, se conserva el último aplicado/cacheado
  y se libera el loader igual (no bloquea la página).
- Importar un esquema valida cada color (isValidColorLocal) y descarta inválidos.
- Reglas: settings/appearance y colorSchemes → write solo Super Admin.

No hay modo claro/oscuro alternable por el usuario final: el esquema de colores
es único y configurable por el Super Admin (se documenta como inexistente, no
como pendiente).

HALLAZGO CORREGIDO
═══════════════════════════════════════════════════════════════════════
F1 [MEDIO — seguridad] — El apply público de colores no validaba el valor.
  El editor valida la entrada (HEX/RGB/HSL en el picker) y el import también,
  y solo el Super Admin puede escribir esquemas (firestore.rules). Pero el punto
  donde un valor de Firestore/caché se vuelve CSS en vivo —
  js/color-scheme.js (keyMapToCssVarMap → setProperty) y
  js/color-scheme-instant.js (applyMap en la primera pintura)— aplicaba CUALQUIER
  string sin validar. Un token usado como `background: var(--x)` con un valor
  url(...) podría cargar una URL; y CSS arbitrario podría colarse por esa vía.
  FIX: se agregó isSafeColorValue() en ambos módulos públicos. Un token solo se
  aplica si su valor es un color ESTRICTO (HEX de 3/4/6/8, rgb(a) o hsl(a),
  anclado ^...$, máx 64 chars). Cualquier otro valor se omite y el token cae a
  su default de color-tokens.css. Así el sitio público NUNCA aplica una URL
  peligrosa ni CSS arbitrario, sin importar cómo haya llegado el valor
  (defensa en profundidad — cumple "no permitas insertar ... URLs peligrosas
  desde el editor" en el punto de enforcement).
  Verificado con casos: acepta #AD3F67, #fff, rgba(20,10,14,0.55), hsl(210 50% 40%);
  rechaza url(https://evil), "red; } body{...}", "rgb(1,2,3) url(x)",
  "expression(...)", "<script>".
