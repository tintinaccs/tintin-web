BUG REAL ENCONTRADO — "0" cuando la consulta no cargó / falló
═══════════════════════════════════════════════════════════════════════
En Estadísticas, los indicadores de pedidos y usuarios ya mostraban "—" cuando
su consulta no estaba lista o fallaba (adminRealtimeReady.orders / .users). Pero
los indicadores derivados de TRÁFICO, PRESENCIA y PRODUCTOS se pintaban con el
conteo crudo, así que ante una lectura fallida o aún en curso mostraban "0":

  - statistics-visitors  → "0"        (siteTraffic no leído/falló)
  - statistics-sessions  → "0 sesiones"
  - statistics-online    → "0"        (sitePresence falló; el Dashboard sí ponía
                                        "—", la sección Estadísticas no)
  - statistics-active-products / statistics-low-stock → "0 con stock bajo"
                                        (products aún sin cargar o con error)
  - statistics-blocked-users → "0 bloqueados" (sin respetar disponibilidad)

Eso viola la regla explícita: "No presentes cero cuando la consulta falló;
diferenciá claramente vacío, error, sin permisos y sin conexión." Un "0" ahí es
indistinguible de "no hay datos" cuando en realidad la consulta no se completó.

CAMBIOS APLICADOS (mínimos y quirúrgicos) — js/admin-app.js
═══════════════════════════════════════════════════════════════════════
1) adminRealtimeReady ahora tiene una bandera por fuente:
   { orders, users, products, traffic, presence }.

2) renderGeneralStatistics: cada indicador derivado se gatea con su bandera y
   cae a "—" si la fuente no resolvió con éxito:
   - visitors / sessions → adminRealtimeReady.traffic
   - conversion          → adminRealtimeReady.orders && .traffic && uniqueVisitors
   - online              → adminRealtimeReady.presence
   - active-products / low-stock → adminRealtimeReady.products
   - blocked-users       → adminRealtimeReady.users

3) Las banderas se encienden al primer éxito y se apagan al fallar, con
   re-render inmediato para que el indicador vuelva a "—" (no quede un valor
   viejo ni un "0"):
   - listenStatisticsTraffic: traffic=false al iniciar, true tras cargar el
     historial, false + renderGeneralStatistics() en el catch.
   - listenDashboardPresence: presence=true en éxito, false + render en error.
   - loadProductos: products=true en éxito, false + render en error.

Diferenciación resultante:
   - Cargando / sin resolver → "—"
   - Error / sin permisos / sin conexión (la lectura lanza) → "—" + estado de
     error visible (statistics-live-status / prod-loading / dashboard-live-status)
   - Vacío real (fuente OK, cero registros) → "0" legítimo

  gasto por usuaria (order-stats: validForSpent = !isCancelled, Math.max(0,…)).
- Ticket promedio = revenue / validOrders.length con guardia de cero y bandera.
- Conversión con guardia de denominador (uniqueVisitors) — nunca divide por 0.

PRIVACIDAD (no IP / no GPS)
- site-activity.js: guarda solo ciudad/región/país/countryCode/geoSource
  aproximados; nunca IP, GPS, coordenadas, código postal, navegador, nombre,
  correo ni referidor. ID aleatorio que ROTA por día (dailyId) y no se vincula a
  la cuenta. Todo bajo consentimiento (hasStatisticsConsent / onPrivacyConsentChange).
- functions/api/visitor-geo.js: devuelve approximate:true con ciudad/región/país;
  "Nunca se devuelve IP, coordenadas, código postal, ASN…".
- firestore.rules: presenceIsValid / trafficSessionIsValid usan hasOnly con la
  lista blanca de campos → el servidor RECHAZA cualquier IP/lat/lng/postal.

SESIONES / PRESENCIA
- Una sesión por pestaña y día (SESSION_RECORDED_PREFIX); heartbeat 60s que se
  detiene con la pestaña oculta o en pagehide; el panel (admin/admin-images) no
  se cuenta como tráfico; localhost/previews tampoco escriben.
- Listeners liberados en pagehide (stopAdminRealtimeData / stopDashboardActivityMetrics).

AUDITORÍA / LOGS
- logAudit registra actor (email+rol), acción, objetivo (tipo/id/label), fecha
  (serverTimestamp) y si fue masiva (bulk/bulkCount).
- firestore.rules: auditLog read solo Super Admin; update/delete: if false
  (inmutable). La tabla diferencia cargando / error / vacío. La única acción
  masiva es exportar CSV (nunca editar/borrar).
- sitePresence/siteTraffic: read/delete solo Super Admin; sesiones de tráfico
  con update: if false (inmutables).

DIAGNÓSTICO
- initSiteDiagnostics se inicia solo para el Super Admin real
  (role === 'superadmin' && email === SUPER_ADMIN). Cobertura de falsos positivos
  y diagnóstico real ya blindada por audit-diagnostics.js / audit-diagnostic-findings.js.

LÍMITES / PAGINACIÓN
- Historial de tráfico con getDocsPaginated (maxDocs 5000). Recalculación global
  de stats corta si supera el tope seguro (truncated → error explícito).
- Listeners de pedidos/usuarios con limit(10000); auditLog con limit(200).
